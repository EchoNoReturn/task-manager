## Context

当前系统仪表盘功能基础，仅展示欢迎信息。缺乏任务统计、日历视图和归档管理能力。任务状态流转无严格控制，删除权限混乱。

本设计涵盖三个主要功能的实现方案：
1. **仪表盘统计**: 提供任务完成数、接取数、进行中数、工作时长等聚合数据
2. **日历视图**: 支持多视图切换和时间线导航，任务可点击跳转
3. **任务归档**: 规范删除/归档行为，支持归档理由和归档任务查询

## Goals / Non-Goals

**Goals:**
- 仪表盘快速展示个人工作绩效指标
- 日历视图直观呈现任务时间分布
- 任务状态单向流转，防止状态回退
- 归档功能保留任务历史，支持合规审计

**Non-Goals:**
- 不支持任务批量归档（单个任务操作）
- 不支持自定义统计周期（固定周/月）
- 不实现任务回收站功能（归档即为终态）

## Decisions

### 1. 仪表盘统计数据提供方式

**决定**: 新增 `DashboardStatsService` 聚合各模块数据，通过 REST API 提供

**理由**: 
- 解耦统计逻辑与业务服务
- 支持未来扩展为实时数据

**替代方案考虑**:
- 直接在 TaskService 中添加统计方法 → 增加服务复杂度
- 前端聚合 → 增加请求次数和数据传输

### 2. 日历视图实现

**决定**: 使用全栈渲染模式，后端提供日期范围查询接口

**理由**:
- 减少前端复杂度，支持 SEO
- 复用现有任务查询逻辑

**替代方案考虑**:
- 前端日历组件 + API 按需加载 → 多次请求，复杂状态管理

### 3. 任务归档数据模型

**决定**: 在 Task entity 添加 `archivedAt`, `archiveReason`, `archivedBy` 字段，不单独建表

**理由**:
- 减少查询复杂度（单表查询归档任务）
- 保留任务完整上下文

**替代方案考虑**:
- 单独归档表 → 任务详情查询需要 JOIN，复杂
- 软删除复用 → 归档有特殊业务语义（理由），不适合复用

### 4. 归档理由必填规则

**决定**: 归档时根据当前状态判断是否必填理由
- 未完成/未关闭状态 → 必填理由
- 已完成/已关闭状态 → 自动记录状态，无需用户输入

**理由**: 完成/关闭状态归档是正常流程，无需额外解释

## Risks / Trade-offs

| Risk | Mitigation |
|------|------------|
| 统计查询性能（大时间范围） | 添加数据库索引，限制查询范围 |
| 日历视图数据量大 | 分页加载，虚拟滚动 |
| 归档后数据不可恢复 | 归档前确认弹窗，理由记录完整 |

## Open Questions
